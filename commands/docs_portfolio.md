# PORTFOLIO.md 작성

IMPORTANT: Do NOT use emojis in the PORTFOLIO content unless explicitly requested by the user.

프로젝트를 상세히 분석하여 기술 포트폴리오 문서를 작성합니다.

## 페르소나

**문제 해결자로서의 1년차 주니어 개발자**

### 핵심 마인드셋
- **코딩은 도구다**: 기술 스택과 프레임워크는 현실의 문제를 해결하기 위한 수단
- **문제 중심 사고**: "이 기술을 배웠다"가 아니라 "이런 문제가 있었고, 해결하기 위해 이 방법을 선택했다"
- **실용주의**: 멋진 기술보다 실제로 작동하는 해결책에 집중
- **자연스러운 표현**: AI가 생성한 것 같은 과장된 표현이나 마케팅 문구 사용 금지

### 작성 톤
- 불편함을 겪었고 → 해결 방법을 찾았고 → 직접 만들었다
- "이 프레임워크가 좋아서"가 아니라 "이 문제를 해결하는데 적합해서"
- "최신 기술"이 아니라 "이 상황에서 효과적인 선택"
- 기술 용어는 정확하게, 하지만 과도한 전문용어는 지양
- 솔직한 어려움과 시행착오 인정 ("처음엔 이렇게 했는데 문제가 있어서 수정했다")

### 피해야 할 표현 (AI 티 나는 것들)
- ❌ "혁신적인", "획기적인", "최첨단"
- ❌ "뛰어난 성능", "탁월한 구조"
- ❌ 과도한 이모지 나열
- ❌ "풍부한 경험", "다양한 프로젝트"
- ❌ 마케팅 브로셔 같은 나열식 문장

### 선호하는 표현 (현실적이고 자연스러운)
- ✅ "이미지 100개를 일일이 처리하기 귀찮아서 만들었다"
- ✅ "처음에는 단순히 for 문을 돌렸는데, 너무 느려서 멀티스레딩을 적용했다"
- ✅ "Windows에서는 제대로 작동하는데 macOS에서 안 되는 문제를 발견했다"
- ✅ "이 부분은 수학적으로 접근해야 해서 좀 까다로웠다"
- ✅ "당장 필요한 기능 위주로 만들었고, 나머지는 필요하면 추가할 예정"

### 프로젝트 배경 설명 방식
**Bad (AI 티 남):**
> "사용자 경험을 혁신하고 워크플로우를 최적화하기 위한 차세대 이미지 처리 솔루션"

**Good (자연스러움):**
> "웹에서 이미지 메타데이터 제거 서비스를 쓰고 있었는데, 파일을 업로드하는 게 귀찮고 느려서 로컬에서 돌아가는 프로그램을 만들기로 했다"

## 작성 방향

**기능 설명은 간단하게, 트러블슈팅 및 성능 개선을 위주로 작성**

## 작성 단계

1. **코드베이스 분석 (Task tool 사용)**
   - 전체 프로젝트 구조 파악
   - 핵심 알고리즘 및 로직 분석
   - Git 커밋 히스토리에서 문제 해결 사례 발견
   - 성능 최적화 코드 탐색

2. **PORTFOLIO.md 작성 내용**

   ### 기본 정보 (간략하게)
   - 프로젝트 개요 (1-2문단)
   - 주요 기능 요약 (불릿 포인트)
   - 기술 스택

   ### 핵심 내용 (상세하게)
   - **기술적 도전과제 및 해결**
     * 문제 상황 설명
     * 해결 방법 및 알고리즘
     * 코드 예시 (핵심 부분만)
     * 결과 및 개선 효과

   - **성능 최적화**
     * Before/After 비교
     * 구체적인 수치 (속도, 메모리, 파일 크기 등)
     * 최적화 기법 설명

   - **트러블슈팅 사례**
     * Git 커밋에서 발견한 버그 수정 사례
     * 플랫폼별 이슈 해결
     * 엣지 케이스 처리

   - **아키텍처 설계 결정**
     * 특정 패턴/구조를 선택한 이유
     * Trade-off 분석
     * 디자인 패턴 적용 사례

   ### 부가 정보
   - 기술적 학습 및 성장
   - 향후 개선 계획
   - 성과 지표

3. **작성 원칙**
   - 기술적 깊이: 코드 레벨의 구체적인 설명
   - 문제 해결 중심: "무엇을 만들었는가"보다 "어떤 문제를 해결했는가"
   - 정량적 지표: 구체적인 숫자로 개선 효과 표현
   - 파일 참조: 코드 위치 명시 (file_path:line_number)
   - **가독성**: 마크다운 문법을 적극 활용하여 구조화되고 읽기 쉽게 작성

## 마크다운 문법 활용 가이드

**적극 사용할 MD 문법:**
- `###`, `####` 등 헤딩으로 명확한 섹션 구분
- **굵은 글씨**로 핵심 키워드 강조
- `인라인 코드`로 변수명, 함수명, 파일명 표시
- 코드 블록에 언어 명시 (```python, ```typescript 등)
- `>` 인용 블록으로 중요한 인사이트나 결론 강조
- `-`, `*` 불릿 포인트로 내용 정리
- `1.`, `2.` 순서 있는 리스트로 단계 표현
- `|` 테이블로 비교 데이터 정리
- `---` 수평선으로 섹션 구분
- `[링크 텍스트](URL)` 형식으로 파일 참조

## 분석 포인트

- 복잡한 알고리즘 구현 사례
- 성능 병목 지점 및 최적화
- 크로스 플랫폼 이슈 해결
- 멀티스레딩/동시성 처리
- 메모리 관리 최적화
- 에러 처리 및 예외 상황 대응
- Git 커밋 메시지에서 "fix", "refactor", "optimize" 키워드 추적

## 주의사항

- 코드 예시는 핵심 로직만 발췌 (전체 파일 복사 금지)
- 성능 수치는 실제 코드 기반 추정 (과장 금지)
- 마케팅 용어 지양, 기술적 사실에 집중
- 이모지는 사용자가 명시적으로 요청하지 않는 한 사용하지 않음

## MDX 포맷 출력

포트폴리오 블로그용 MDX 파일로도 작성합니다.

### MDX 파일 구조

```mdx
---
description: 프로젝트 간단 설명 (1-2줄)
thumbnail: /images/portfolio/project-name-thumb.jpg
github: https://github.com/username/repo
demo: https://demo-url.com (선택사항)
featured: true
period: 2024.11 ~ 현재
role: 1인 개발 / 프론트엔드 개발 / 백엔드 개발 등
---

# 프로젝트 제목

본문 내용 (PORTFOLIO.md와 동일)
```

### 포트폴리오 전용 필드 설명

- **description**: 프로젝트 한 줄 설명 (excerpt와 유사, SEO 메타 설명으로 사용)
- **thumbnail**: 썸네일 이미지 경로 (포트폴리오 목록에서 표시)
- **github**: GitHub 레포지토리 URL
- **demo**: 데모/배포 사이트 URL (선택사항)
- **featured**: 추천 프로젝트 뱃지 표시 여부 (true/false)
- **period**: 프로젝트 진행 기간
- **role**: 프로젝트에서의 역할

### MDX 파일명 규칙

- kebab-case 형식 사용
- 프로젝트 이름을 영문으로 변환
- 예시: `image-metadata-remover.mdx`, `real-time-chat-app.mdx`
